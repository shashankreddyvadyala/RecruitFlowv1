INVOICE TAXABILITY CLASSIFICATION — FULL ARCHITECTURE
================================================================================

ARCHITECTURE APPROACH: HYBRID
  ├── Databricks (Phases 1-3): Batch classification pipeline
  │     Spark jobs + Foundation Model APIs + MLflow — all compute runs

================================================================================
PHASE 3: BATCH CLASSIFICATION PIPELINE (Databricks — Spark Notebook)
================================================================================

Input:  atims_shared.staging.prediction_ready (Delta table from Phase 2)
Output: atims_prod.ml_output.non_norad_predictions (Delta table)

Runtime: Databricks Workflow Task (runs after Phase 2 completes)
Cluster:  Standard compute (GPU not required for inference)

Rationale for Databricks (not AAW/LangGraph):
  - All data, models, and knowledge bases already live in Databricks
  - Eliminates 100K+ cross-service API calls per month
  - Spark parallelism >> asyncio.gather for batch throughput
  - Native MLflow integration for full traceability
  - Foundation Model APIs (ai_query, ai_classify) available natively
  - Batch-level QA context (full month's data in one DataFrame)
  - Estimated cost reduction: ~50-60% vs AAW-hosted approach

Databricks AI Capabilities Used:
  - ai_classify()        : SQL function for zero-shot LLM classification
  - ai_query()           : SQL function for Foundation Model API calls
  - Vector Search SDK    : Batch similarity search (Rules KB, Write-ups KB)
  - Model Serving        : XGBoost endpoint (xgboost-non-norad)
  - Foundation Model API : GPT-4o for orchestrator LLM resolution
  - MLflow               : Experiment tracking + model registry

Notebook/Task Structure:
  ┌─────────────────────────────────────────────────────────────────┐
  │  TASK 3.1: Rules Expert                                         │
  │                                                                 │
  │  Action 1: Deterministic lookup (Spark SQL JOIN)                │
  │    SELECT * FROM prediction_ready r                             │
  │    JOIN deterministic_rules d                                   │
  │      ON r.extc=d.extc AND r.mic_cln=d.mic AND                  │
  │         r.account_code=d.account_code                           │
  │    → rules_matched_df (category + confidence=1.0)               │
  │    → no_rules_df (left anti join)                               │
  │                                                                 │
  │  Action 2: Vector search fallback on no_rules_df                │
  │    vsc = VectorSearchClient()                                   │
  │    index = vsc.get_index("rules_kb_index")                      │
  │    Batch similarity search on final_description                 │
  │    → rag_matched_df (strong match: category + rule_id)          │
  │    → unmatched_df (no match)                                    │
  │                                                                 │
  │  Output columns added: rule_match, rule_id, category,           │
  │                         confidence, match_type                  │
  └─────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────┐
  │  TASK 3.2: Classifier (on unmatched_df only)                    │
  │                                                                 │
  │  Sub-model A: XGBoost (batch predict via model serving)         │
  │    client = get_deploy_client("databricks")                     │
  │    preds = client.predict("xgboost-non-norad",                  │
  │                           {"inputs": embeddings_list})          │
  │    → xgboost_pred, xgboost_conf per record                      │
  │                                                                 │
  │  Sub-model B: LLM zero-shot via ai_classify()                   │
  │    SELECT *, ai_classify(final_description,                     │
  │      ARRAY('Cable - Metallic', 'Software License', ...))        │
  │      AS llm_category FROM unmatched_temp                        │
  │    → gemma_pred, gemma_conf per record                          │
  │                                                                 │
  │  Agreement Logic (Spark DataFrame operations):                  │
  │    Both agree + conf > 0.8 → ACCEPT (final_category set)        │
  │    Both agree + low conf  → MEDIUM confidence                   │
  │    Disagree               → orchestrator_df (LLM resolution)   │
  │                                                                 │
  │  LLM Resolution (on orchestrator_df, ~10-15% of unmatched):     │
  │    ai_query('databricks-gpt-4o',                                │
  │      CONCAT('XGBoost predicted: ', xgboost_pred,                │
  │             ', LLM predicted: ', llm_category,                  │
  │             ', Description: ', final_description,               │
  │             '. Resolve to best category from: ...'))            │
  │                                                                 │
  │  Output columns: xgboost_pred, xgboost_conf, llm_pred,          │
  │                  llm_conf, agreement, final_category,           │
  │                  ensemble_confidence                            │
  └─────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────┐
  │  TASK 3.3: Tax Lookup (pure SQL JOIN)                           │
  │                                                                 │
  │  SELECT r.*, m.taxability                                       │
  │  FROM classified_df r                                           │
  │  LEFT JOIN atims_shared.reference.taxability_matrix m           │
  │    ON r.final_category = m.cluster                              │
  │    AND r.state = m.state                                        │
  │                                                                 │
  │  Note: Tax Reasoning agent removed. Records with no             │
  │  matrix match receive taxability=NULL and are flagged           │
  │  for human review (review_priority escalated to HIGH).          │
  │                                                                 │
  │  Output columns: taxability ("Taxable"|"Exempt"|null),          │
  │                  tax_source ("matrix"|"no_match")               │
  └─────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────┐
  │  TASK 3.3b: Use Tax Lookup (Spark DataFrame operation)          │
  │                                                                 │
  │  Evaluates whether a use tax reversal applies to each record.   │
  │  Determination is based solely on the use_tax_amount input      │
  │  column sourced from the upstream AP system.                    │
  │                                                                 │
  │  Logic (Spark withColumn):                                      │
  │    CASE                                                         │
  │      WHEN use_tax_amount > 0 AND taxability = 'Exempt'          │
  │        THEN 'Yes'   -- use tax paid but invoice is exempt;      │
  │                        reversal required                        │
  │      WHEN use_tax_amount > 0 AND taxability = 'Taxable'         │
  │        THEN 'No'    -- use tax paid; no reversal needed         │
  │      WHEN use_tax_amount = 0                                    │
  │        THEN 'No'    -- no use tax present                       │
  │      WHEN use_tax_amount IS NULL                                │
  │        THEN NULL    -- DEFERRED: pending client clarification   │
  │                        (NULL = no use tax vs. data gap?)        │
  │      WHEN taxability IS NULL                                    │
  │        THEN NULL    -- taxability unresolved; cannot determine  │
  │    END AS Reverse_UseTax                                        │
  │                                                                 │
  │  Input columns consumed: use_tax_amount (number), taxability    │
  │  Output column added:    Reverse_UseTax ('Yes'|'No'|null)       │
  │  Open question: confirm NULL use_tax_amount treatment with      │
  │    client before finalising this branch                        │
  └─────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────┐
  │  TASK 3.4: Tax Writer (batch LLM via ai_query())                │
  │                                                                 │
  │  FAST path (matrix hit, ~85%): template-based write-up          │
  │    RAG: Vector Search on Prior Write-ups KB (batch)             │
  │    ai_query('databricks-gpt-4o',                                │
  │      CONCAT('Generate audit-ready taxability justification.     │
  │             Category: ', final_category, ', State: ', state,    │
  │             ', Taxability: ', taxability,                       │
  │             ', Reverse_UseTax: ', Reverse_UseTax,              │
  │             '. Prior examples: ', retrieved_writeups))          │
  │                                                                 │
  │  Note: When Reverse_UseTax = 'Yes', the write-up must           │
  │  explicitly state that use tax reversal is required and         │
  │  include use_tax_amount in the justification.                   │
  │                                                                 │
  │  Output columns: short_justification, full_writeup,             │
  │                  reviewer_note, audit_ready                     │
  └─────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────┐
  │  TASK 3.5: QA (Spark DataFrame operations + window functions)   │
  │                                                                 │
  │  Check 1: Vendor-category consistency                           │
  │    Window: partitionBy(vendor_id), count category freq          │
  │    Flag if current category != modal category for vendor        │
  │                                                                 │
  │  Check 2: Amount reasonableness                                 │
  │    Window: partitionBy(final_category)                          │
  │    Z-score on amount vs category historical distribution        │
  │                                                                 │
  │  Check 3: State-category history                                │
  │    JOIN against historical_stats table                          │
  │    Flag new combinations not seen in prior 12 months            │
  │                                                                 │
  │  Check 4: Cross-validation                                      │
  │    Flag if rule_match=true but classifier agrees on             │
  │    different category (conflict between rule and ML)            │
  │                                                                 │
  │  Check 5: Taxability sanity                                     │
  │    JOIN against state_service_tax_rules table                   │
  │    Flag if Service taxable in no-service-tax state              │
  │                                                                 │
  │  Check 6: Batch-level outlier detection (NEW vs AAW approach)   │
  │    Flag records where category differs from 95%+ of same-       │
  │    vendor records in current month's batch                      │
  │                                                                 │
  │  Output columns: anomalies (array), anomaly_score (0.0-1.0),    │
  │                  review_priority (LOW/MED/HIGH/CRITICAL),       │
  │                  override_suggested, recommendation             │
  └─────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────┐
  │  TASK 3.6: Re-route (iterate on flagged records, max 2 passes)  │
  │                                                                 │
  │  Filter: WHERE override_suggested = true AND iteration < 2      │
  │  Re-run TASK 3.2 (Classifier) with QA hint injected into prompt │
  │  Update iteration count, re-run TASKS 3.3, 3.3b, 3.4, 3.5      │
  │  Remaining anomalies after 2 iterations → ESCALATION path       │
  └─────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────┐
  │  TASK 3.7: Write Final Output                                   │
  │                                                                 │
  │  Merge all DataFrames (rules_matched + rag_matched + classified) │
  │  Write to: atims_prod.ml_output.non_norad_predictions           │
  │  Trigger: POST webhook to FastAPI backend                        │
  │           {month, records_processed, flagged_count}             │
  └─────────────────────────────────────────────────────────────────┘

PROCESSING PATHS SUMMARY
================================================================================

  Note: All paths are now Databricks Spark batch operations (not per-record
  LangGraph executions). "Latency" below refers to per-record effective
  processing time within the batch job, not wall-clock end-to-end.

  FAST PATH (est. 60% of records):
    Rules Expert (deterministic SQL match) → Tax Lookup (matrix JOIN)
    → Tax Writer (ai_query) → QA (window functions, pass) → Output
    Cost: ~$0.004/record  (SQL + one ai_query call)
    Effective latency: <1 second

  STANDARD PATH (est. 25% of records):
    Rules Expert (no match) → Classifier (XGBoost + ai_classify, agree)
    → Tax Lookup (matrix JOIN) → Tax Writer (ai_query) → QA (pass) → Output
    Cost: ~$0.006/record  (XGBoost endpoint + ai_classify + ai_query)
    Effective latency: ~1-2 seconds

  COMPLEX PATH (est. 10% of records):
    Rules Expert (no match) → Classifier (disagree)
    → LLM resolution (ai_query) → Tax Lookup → Tax Writer → QA (pass) → Output
    Cost: ~$0.015/record  (two ai_query calls)
    Effective latency: ~2-4 seconds

  ESCALATION PATH (est. 5% of records):
    Full pipeline → QA flags anomaly → Re-route (max 2 passes)
    → Still uncertain or null taxability → Flagged for human review
    Cost: ~$0.020/record + human review time
    Latency: Async (human in loop via Reviewer Agent)

MODEL COMPARISON & SELECTION RATIONALE
================================================================================

  EMBEDDINGS:
    Selected: databricks-gte-large-en (1024-dim)
    Replaces: TF-IDF + PCA (NON-NORAD) and BertSentenceEmbeddings + PCA (NORAD)
    Used in: Phase 2 (generate + store), Phase 3 (XGBoost input + Vector Search)
    Why: Dense semantic vectors, no PCA needed, pre-deployed on Databricks

  BATCH CLASSIFIER A: XGBoost
    Endpoint: xgboost-non-norad (Databricks Model Serving)
    Input: gte_embedding (1024-dim)
    Called from: Phase 3 Task 3.2 (batch serving call)
    Why: Fast, deterministic, no per-call LLM cost

  BATCH CLASSIFIER B: LLM zero-shot via ai_classify()
    Function: Databricks ai_classify() SQL function
    Replaces: databricks-gemma-3-12b endpoint call
    Why: Native Spark SQL, no endpoint management, billed per token
         Equivalent zero-shot accuracy to Gemma 3 12B

  LLM RESOLUTION + TAX WRITER (Phase 3):
    Function: ai_query() SQL function → GPT-4o
    Used for: Classifier disagreement resolution (~10-15% of unmatched)
              Tax Writer write-up generation (all records)
    Why: Native Databricks Foundation Model API, no cross-service calls

  UI AGENTS (Phases 4-5):
    Model: GPT-4o via ChatLiteLLM (AAW CDO proxy)
    Used by: Reviewer Agent, Feedback Agent, Labeling Agent
    Why: All agent models unified to GPT-4o via AAW; no Gemma/Claude/Opus

  TAX REASONING AGENT: REMOVED
    Reason: Records with no taxability matrix match are escalated directly
    to human review (review_priority = HIGH). The added complexity and
    LLM cost of reasoning from raw tax law was not justified.

  FINE-TUNING OPTION (future consideration):
    Candidate: Fine-tune a smaller model on labeled invoice data
    Accuracy: ~88-93% (would allow dropping XGBoost entirely)
    Trade-off: Requires ~1-2 weeks setup + monthly retraining
    If pursued: Replace both ai_classify() + XGBoost with single model
